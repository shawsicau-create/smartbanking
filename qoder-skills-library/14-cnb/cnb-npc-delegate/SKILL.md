# 召唤 NPC 协同工作

当遇到以下情况时，应主动召唤其他 NPC：

1. **任务超出能力范围**：需要特定领域专家（如前端、测试、安全等）
2. **任务可并行处理**：多个子任务可同时进行
3. **需要跨仓库协作**：涉及其他仓库的代码或功能
4. **其他 NPC 更擅长**：效率更高

## 召唤流程

### 1. 评估任务，决定是否召唤

问自己：
- 这个任务是否需要特定领域的专业知识？
- 是否有其他 NPC 更擅长处理这个任务？
- 任务是否可以拆解并并行处理？

### 2. 搜索合适的 NPC

使用 `cnb-npc-search` skill 搜索：

```bash
cnb knowledge-base search-npc --key "关键词" --top-n 3
```

示例场景对应搜索词：
| 场景 | 搜索关键词 |
|---|-----------|
| 需要代码审查 | `代码审查`、`CR` |
| 需要写前端 | `前端开发`、`React` |
| 需要写测试 | `单元测试`、`测试用例` |
| 需要性能优化 | `性能优化`、`调优` |
| 需要安全审计 | `安全`、`审计` |

### 3. 正确发起召唤

找到合适的 NPC 后，使用正确的 @ 格式召唤：

```
@{slug}({name})
```

示例：
- `@npc/hok(π)` - 仓库 `npc/hok` 下名为 `π` 的 NPC
- `@npc/code-reviewer(代码审查员)` - 仓库 `npc/code-reviewer`

### 4. 传递完整的上下文

召唤时提供充分信息：
- **任务背景**：要解决的问题
- **相关文件**：涉及的代码或路径
- **约束条件**：技术栈、代码规范等
- **期望结果**：交付物形式

示例召唤方式：

```
@npc/xxx 帮我审查 /workspace/src/auth 目录下的登录逻辑，重点关注：
1. 密码存储是否安全
2. Session 管理是否有漏洞
参考标准：OWASP Top 10
```

## 协同工作模式

召唤成功后，可：

1. **分工协作**：拆分任务，各 NPC 处理一部分
2. **串联审核**：一个 NPC 完成后，另一个继续优化
3. **并行处理**：多个 NPC 同时处理独立任务

## 何时主动建议召唤

作为 AI，以下情况应主动建议用户召唤其他 NPC：

- 需求明显超出当前 NPC 能力
- 任务涉及多个专业领域
- 任务耗时长，可拆解并行

## 注意事项

- 召唤前先了解目标 NPC 的能力范围（通过 slogan 判断）
- 尽量一次召唤到位，避免反复补充上下文
- 模糊需求先自行分析，再决定是否召唤
- 始终把控整体任务，即使分配了子任务
- 评论中如果只提及 npc 不召唤，应该去掉 `@` 符号，或使用反引号包裹 `@npc`
